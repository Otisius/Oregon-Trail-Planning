Unity architecture rules
The golden rule: never let two systems talk directly. Everything goes through the EventBus or a manager. This is what makes multiplayer survivable when you add it later.

Principle 1
EventBus — never direct references between systems
Bad
SupplySystem.instance.food -= 10 called from inside WagonController.
Good
EventBus.Publish(new SupplyEvent(-10, "food")). Later a NetworkSupplySync just subscribes. Zero refactoring.
Principle 2
Managers own state — components own behaviour
SupplySystem
Owns all resource values via ModifySupply(). Never touches UI. Later values become NetworkVariable<float>.
WagonController
Owns movement only. Reads from PlayerInput, never Input.GetKey(). Later you add NetworkTransform and it works.
PartySystem
Plain C# classes, no MonoBehaviours. Later becomes a synced list without structural changes.
Avoid
Things that will hurt you later
Input.GetKey() anywhere
Remote players don't have a keyboard. Use Unity's PlayerInput + Input Actions from day one.
Game logic in UI scripts
UI reads state, never owns it. If your HUD is calculating supply drain you'll regret it.
FindObjectOfType<T>() at runtime
Use GameManager.Instance.SupplySystem as your service locator instead.
Unreal architecture rules
Unity → Unreal mental model: everything has a direct equivalent, just different names. Learn the mapping before you start.
Concept map
Unity → Unreal equivalents
GameManager singleton → Game Instance
Set your Blueprint Game Instance in Project Settings. Accessible anywhere via Get Game Instance → Cast. Persists across level loads automatically.
ScriptableObject → Data Asset
Blueprint class inheriting from Data Asset. Define fields, edit instances in the content browser. Identical workflow.
EventBus → Event Dispatcher
Defined on the broadcasting Blueprint. Others bind on BeginPlay. Broadcast fires all listeners. Same pattern, different syntax.
Cinemachine → Spring Arm + Camera
Spring Arm handles follow cameras with lag and collision. Sequencer handles cutscenes. Both Blueprint-native, no plugins needed.
NGO replication → Blueprint replication
No library. Check Replicated on Blueprint variables. Use Server / Multicast / Client event types. UE handles the rest.
Avoid
Unreal-specific traps
Baked lighting
You're using Lumen — it's fully dynamic. Baking breaks the whole point and makes iteration painfully slow.
Storing runtime state in Data Assets
Data Assets are config, not state. Mutable values like current food live in the Game Instance, not the asset.
Get Player Controller for game logic
In multiplayer it's server-relative and unreliable on clients. Use Game State and Game Instance instead.
Blueprint is totally fine
Blueprint handles everything in this game. Same separation rules as Unity — state in managers, behaviour in components, communication through dispatchers.
